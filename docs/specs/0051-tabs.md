# 0051 - Tabs

## Problem

Canopy has containers for one surface at a time - `Dialog` (0019) and `ResponsiveDialog` for
modal surfaces, `SideNav` / `TopNav` for app navigation - but nothing for the everyday pattern of
**switching between sibling panels in place**: settings sections, an entity's overview/activity/
files, a form split into stages. Today a consumer who needs tabs has to hand-roll the roving
`tablist` / `tab` / `tabpanel` wiring - arrow-key navigation, `aria-selected`, focus management,
the active-underline styling - re-implementing the same accessible switcher every time. That is
exactly the reusable interaction the design system should own once.

shadcn ships a Tabs on `@radix-ui/react-tabs`, but canopy has no equivalent and does not yet
depend on that primitive. This is for any content-dense surface that groups related views a user
toggles between without leaving the page: profile/settings screens, detail panes, dashboard
sections. It sits alongside the other stateful `Branch` components (`Dialog`, `Combobox`,
`SideNav`) - it owns interaction state (the selected tab, roving focus) but no portal.

## Outcome

- A new canopy component family, `Tabs`, exported from `@rogueoak/canopy/branches`, built on
  `@radix-ui/react-tabs`, with four parts that compose like the other Branch families:
  `Tabs` (root), `TabsList`, `TabsTrigger`, `TabsContent`.
- **Selection**: works both **uncontrolled** (`defaultValue`) and **controlled** (`value` +
  `onValueChange`); the active trigger carries `data-state="active"` and shows the canopy active
  styling (an underline for horizontal, a pill option documented in Approach), inactive triggers
  read muted.
- **Keyboard + a11y**: correct roles - `tablist` on the list, `tab` on each trigger, `tabpanel`
  on each content region - with roving focus and Radix's arrow-key navigation (Left/Right for
  horizontal, Up/Down for vertical), `Home`/`End`, and `aria-selected` / `aria-controls` /
  `aria-orientation` wired by the primitive. A `disabled` trigger is skipped by roving focus and
  is not selectable.
- **Orientation**: `orientation="horizontal"` (default) and `orientation="vertical"`, changing the
  list layout and the arrow keys that move between tabs.
- **Theming**: styled with the 0005 recipe (full literal semantic-token Tailwind utilities,
  `cn()` merge, `forwardRef` + native prop spread on every styled part), so it themes light/dark
  through the token layer with no `dark:` on the common path.
- **Docs**: a Storybook catalog entry (playground, orientations, disabled tab, controlled,
  many-tabs) rendering in both themes; canopy README component list and the `overview/` living
  docs updated on completion.

### Out
- **Automatic / async tab loading** (lazy-mounting panels only when first activated, suspense
  boundaries) - v1 mounts panels per Radix defaults; a lazy-mount follow-up can add a
  `forceMount` / lazy prop later.
- **Scrollable / overflowing tab strips** (arrow-scroll affordances, "more" overflow menu) for
  very long tab lists - v1 lets the list wrap/scroll natively; an overflow-aware `TabsList` is a
  clean follow-up.
- **Animated panel/indicator transitions** (sliding underline, cross-fade panels) - v1 swaps
  panels instantly; motion can be added later behind `motion-reduce:` guards.
- **Nested / URL-synced tabs** (routing integration) - left to the consumer via the controlled
  API.
- Changing any unrelated component - `Tabs` is purely additive; `Dialog`, `Combobox`, `SideNav`,
  and the Seeds/Twigs are untouched.

## Approach

**Primitive stack: `@radix-ui/react-tabs` with the 0005 recipe.** Canopy is built entirely on
Radix primitives; Tabs follows suit. `@radix-ui/react-tabs` supplies the whole accessible
machine - roving tabindex, arrow/Home/End navigation, `orientation` handling, and the
`tablist`/`tab`/`tabpanel` roles with `aria-selected` / `aria-controls` / `aria-orientation`
wired between parts - so canopy only styles it. There is no portal (unlike `Dialog`/`Combobox`),
so no theming caveat: the whole tree renders in place and inherits `.dark` from `<html>` like any
in-page component. `@radix-ui/react-tabs` is added as a runtime **dependency** and externalized in
tsup exactly like the other Radix deps, per the canopy externalization rule.

**Part surface (mirrors the shadcn tabs, canopy-styled).** Four `forwardRef` wrappers, each a thin
canopy-styled pass-through over the matching Radix part with `React.ComponentRef` typing and a
full native prop spread:
- `Tabs` - the stateful root over `TabsPrimitive.Root`. Passes through `value` / `defaultValue` /
  `onValueChange` / `orientation` and standard props; owns the selected-tab state.
- `TabsList` - `TabsPrimitive.List` (`role="tablist"`): an inline flex row of triggers for
  horizontal, a flex column for vertical, with a bottom (horizontal) / trailing (vertical)
  `border-border` rail that the active underline sits on.
- `TabsTrigger` - `TabsPrimitive.Trigger` (`role="tab"`): the tab button, `text-label`,
  `text-text-muted` when inactive and `text-text` with an active underline via the
  `data-[state=active]:` variant when selected; the shared `focus-visible:` ring token set;
  `disabled:opacity-50 disabled:cursor-not-allowed` for a disabled tab (toggle-style disabled per
  the recipe, since the tab has no fill to preserve). `cursor-pointer` on the base, matching the
  Button idiom.
- `TabsContent` - `TabsPrimitive.Content` (`role="tabpanel"`): the panel region, with the
  focus-visible ring so the panel is reachable/visible when focused after selection.

**Key decisions & trade-offs.**
- *Underline vs pill styling*: v1 ships the **underline** active style as the default (the
  common, low-chrome tab look and the shadcn default). A `pill` visual is an easy cva variant but
  is deferred to keep the first surface small unless a design need lands; noted so it is not read
  as an omission. If added it must stay full-literal token classes (no dynamic class names).
- *New dependency `@radix-ui/react-tabs`*: one more Radix runtime dep on canopy. It is small,
  first-party to Radix, and exactly the missing primitive; hand-rolling roving focus + ARIA would
  be more code to own and a common source of a11y bugs. Flagged for the **security / architecture
  personas** to weigh the new-dependency surface in review, consistent with how 0030 added
  `@radix-ui/react-popover`.
- *Data-state styling over a custom active prop*: active/inactive is driven by Radix's
  `data-state="active"` through Tailwind `data-[state=active]:` variants (no custom prop, no
  imperative class toggling), matching the aria/data-driven styling learning.

**Styling & recipe.** FULL LITERAL token utility strings so Tailwind v4's scanner emits each,
`cn()` merge with the caller's `className` last (caller wins), `forwardRef` on every styled part
with a native prop spread, `React.ComponentRef<typeof TabsPrimitive.*>` ref typing, semantic
tokens only, and **no `dark:` on the common path** - identical to 0005/0030. `displayName` set
from the Radix part on each wrapper.

**Accessibility.** Radix supplies the `tablist`/`tab`/`tabpanel` roles, roving tabindex,
arrow/Home/End navigation, `aria-selected`, `aria-controls`, and `aria-orientation`. Canopy adds
only styling and does not override the semantics. The a11y promises are guarded by **observable
tests** (roles present, click and arrow keys change the selected tab and revealed panel, disabled
trigger not selectable), per the repo learning that a11y is proven by outcomes, not by asserting a
class exists.

**Motion.** None in v1 (panels swap instantly). Any future indicator/panel animation is added as
a token-layer keyframe gated behind `motion-reduce:animate-none`, not inline.
